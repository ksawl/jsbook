## |1| **XSS (Cross-Site Scripting)**

Механика атаки: Злоумышленник внедряет вредоносный скрипт на страницу, которую просматривают другие пользователи. Скрипт выполняется в контексте их браузера и может украсть куки, токены или данные форм.

### **Типы XSS:**
- **Stored (Хранимая)**: Скрипт сохраняется в базе данных (например, в комментарии или профиле).
- **Reflected (Отраженная)**: Скрипт передается через URL-параметры и "отражается" сервером в ответе.

### **Защита:**
1. **Экранирование (Sanitization)**: Превращайте `<script>` в `&lt;script&gt;`. Современные фреймворки (React, Vue) делают это по умолчанию.
2. **CSP (Content Security Policy)**: Заголовок, ограничивающий источники, с которых разрешено загружать и выполнять скрипты.
3. **HttpOnly Cookies**: Защита кук от чтения через `document.cookie`.

---

## |2| **CSRF (Cross-Site Request Forgery)**

Механика атаки: Злоумышленник заставляет авторизованного пользователя отправить запрос на сервер от своего имени (например, через скрытую форму на вредоносном сайте или картинку `<img src="http://bank.com/transfer?to=hacker">`).

### **Защита:**
1. **Anti-CSRF Tokens**: Сервер выдает уникальный токен для каждой формы, который должен вернуться в теле запроса. Хакер не может прочитать этот токен на другом домене.
2. **SameSite Cookies**: Флаг `SameSite: Strict` или `Lax` запрещает браузеру отправлять куки при кросс-доменных запросах.

---

## |3| **Инъекции: SQL и NoSQL**

Механика атаки: Злоумышленник передает вредоносный код в поля ввода, который затем выполняется базой данных (например, `' OR 1=1 --`).

### **SQL-инъекции:**
Происходят при конкатенации строк в SQL-запросах.
**Защита**: Используйте параметризованные запросы (Prepared Statements). Никогда не вставляйте ввод пользователя напрямую в строку запроса.

### **NoSQL-инъекции (MongoDB):**
Вместо строк могут передаваться объекты-операторы.
```javascript
// Опасно:
User.find({ username: req.body.username, password: req.body.password });
// Если передать password: { $gt: "" }, авторизация пройдет успешно без пароля!
```
**Защита**: Валидация типов данных (например, через `Joi` или `Zod`) и использование ODM (Mongoose), который ограничивает типы по схеме.

---

## **Итог**

- Экранируйте любой ввод пользователя перед выводом на фронтенде.
- Используйте **SameSite** для кук и **CSP** для защиты от скриптов.
- Никогда не доверяйте данным из `req.body` или `req.query` без предварительной валидации.
- Регулярно обновляйте зависимости — в них часто находят уязвимости.

## **Практика**

### 1. **Анализ уязвимости**
Представьте страницу профиля, которая выводит имя пользователя: `<div>Привет, ${user.name}</div>`. Как злоумышленник может использовать это для кражи кук? Напишите пример значения для `user.name`.

### 2. **Защита NoSQL**
Напишите пример кода для Express, который проверяет, что `req.body.username` и `req.body.password` являются именно строками, а не объектами, используя оператор `typeof`.

### 3. **Настройка CSP**
Создайте пример заголовка `Content-Security-Policy`, который разрешает загрузку скриптов только с вашего собственного домена и запрещает выполнение инлайн-скриптов (`eval` и скрипты в атрибутах).
