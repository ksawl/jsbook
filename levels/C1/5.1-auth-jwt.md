## |1| **Что такое JWT?**

**JWT** — это открытый стандарт для передачи информации между сторонами в виде объекта JSON. Эта информация проверяется и является доверенной, так как она имеет цифровую подпись.

### **Зачем это нужно?**
В отличие от сессий на основе куки, где сервер хранит состояние (stateful), JWT позволяет серверу быть **stateless**. Вся необходимая информация о пользователе "запакована" в сам токен. Серверу достаточно проверить подпись, чтобы убедиться, что данные не были подделаны.

---

## |2| **Структура токена**

JWT состоит из трех частей, разделенных точками: `header.payload.signature`.

### **1. Header (Заголовок)**
Обычно содержит тип токена и алгоритм подписи (например, HS256 или RS256).

### **2. Payload (Полезная нагрузка)**
Содержит "заявления" (claims) — данные о пользователе: `id`, `role`, время истечения `exp` и т.д.
> [!WARNING]
> Данные в Payload просто закодированы в Base64. Никогда не кладите в токен пароли или другие секретные данные!

### **3. Signature (Подпись)**
Создается путем кодирования заголовка и нагрузки с использованием секретного ключа. Это гарантирует, что отправитель JWT тот, за кого себя выдает.

## |3| **Cookies vs LocalStorage**

Где хранить токен на клиенте — один из самых жарких споров в вебе.

| Место хранения | Плюсы | Минусы |
| :--- | :--- | :--- |
| **LocalStorage** | Легко реализовать, доступно через JS. | Уязвимо для **XSS**-атак (скрипт может украсть токен). |
| **HttpOnly Cookies** | Защищено от чтения через JS (XSS не страшны). | Уязвимо для **CSRF** (нужна доп. защита). |

> [!IMPORTANT]
> На уровне C1 стандартом считается использование **HttpOnly Cookies** для хранения токенов, так как XSS-атаки более критичны и сложны в предотвращении.

## |4| **Refresh Tokens и безопасность**

Для минимизации ущерба в случае кражи, токены делят на два вида:

1.  **Access Token**: Короткое время жизни (15–60 мин). Используется для доступа к API.
2.  **Refresh Token**: Длинное время жизни (дни/недели). Используется только для получения нового Access Token.

### **Сценарий работы:**
1. Пользователь логинится -> получает Access и Refresh токены.
2. Access Token отправляется с каждым запросом.
3. Когда Access Token просрочен, клиент отправляет Refresh Token на специальный эндпоинт `/refresh`.
4. Сервер проверяет Refresh Token и выдает новую пару.

---

## **Итог**

- **JWT** позволяет серверу не хранить сессии в базе данных (Stateless).
- Токен состоит из заголовка, данных и подписи.
- Храните секреты в **HttpOnly Cookies**, чтобы защититься от кражи через JS.
- Используйте связку **Access + Refresh**, чтобы обеспечить безопасность и удобство пользователя.

## **Практика**

### 1. **Анализ токена**
Возьмите любой JWT (можете сгенерировать на `jwt.io`) и вручную декодируйте его часть Payload с помощью `atob()` в консоли браузера. Какие поля вы там видите?

### 2. **Middleware проверки**
Напишите набросок функции (Middleware) для Express, которая:
1. Достает токен из заголовка `Authorization`.
2. Проверяет его валидность (представьте, что у вас есть функция `verifyToken(token)`).
3. При ошибке возвращает статус `401 Unauthorized`.

### 3. **Безопасное хранение**
Объясните своими словами, почему флаг `SameSite: 'Strict'` в куках помогает защититься от CSRF атак при использовании JWT.
