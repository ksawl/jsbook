## |1| **Архитектура Event Loop**

Event Loop — это бесконечный цикл, который ждет появления задач в очередях и перемещает их в Call Stack (стек вызовов) для выполнения, когда тот пуст.

**Основные компоненты:**
- **Call Stack**: Стек, где выполняется текущий код.
- **Web/C++ APIs**: Фоновые потоки (в браузере — браузерное окружение, в Node.js — LibUV), где выполняются таймеры, запросы и операции с файлами.
- **Task Queues**: Очереди, куда попадают результаты фоновых операций.

---

## |2| **Микрозадачи и Макрозадачи**

Это критически важная концепция для понимания порядка выполнения кода. Очереди имеют разный приоритет.

### **1. Microtasks (Микрозадачи)**
- `Promise.then/catch/finally`
- `queueMicrotask()`
- `MutationObserver` (в браузере)
- `process.nextTick()` (в Node.js, имеет наивысший приоритет)

**Правило**: Event Loop выполняет **все** микрозадачи в очереди сразу после завершения текущей синхронной задачи, **прежде** чем перейти к следующей макрозадаче или отрисовке.

### **2. Macrotasks (Макрозадачи)**
- `setTimeout`, `setInterval`
- `setImmediate` (в Node.js)
- События (click, load)
- I/O операции

---

## |3| **LibUV и фазы цикла в Node.js**

В Node.js событийный цикл реализован библиотекой **LibUV**. Его цикл сложнее браузерного и состоит из фаз:

1.  **Timers**: Выполняются колбэки от `setTimeout` и `setInterval`.
2.  **Pending Callbacks**: Выполняются системные ошибки (например, TCP ошибки).
3.  **Poll (Опрос)**: Самая важная фаза. Здесь Node.js ждет новые I/O события.
4.  **Check (Проверка)**: Выполняются колбэки `setImmediate`.
5.  **Close Callbacks**: Выполняются события закрытия (`socket.on('close', ...)`).

> [!IMPORTANT]
> Между каждой фазой Node.js проверяет очередь микрозадач и выполняет их, если они есть.

---

## |4| **Опасности блокировки**

Если в основном потоке выполняется тяжелый синхронный код (например, огромный цикл или сложная математика), Event Loop не может перейти к следующей фазе. Это называется **Event Loop Lag**.

**Как избежать:**
- Разбивать тяжелые задачи через `setTimeout(0)` или `setImmediate`.
- Выносить тяжелые вычисления в `Worker Threads`.

---

## **Итог**

- **Сначала синхронный код**, затем **все микрозадачи**, затем **одна макрозадача**.
- В браузере отрисовка происходит **между** макрозадачами.
- `process.nextTick` в Node.js выполняется даже раньше, чем микрозадачи промисов.
- Никогда не блокируйте цикл тяжелыми вычислениями: страдают таймеры и ввод-вывод.

## **Практика**

### 1. **Предсказание вывода**
Опишите порядок вывода в консоль для следующего кода и объясните "почему":
```javascript
console.log('1');
setTimeout(() => console.log('2'), 0);
Promise.resolve().then(() => console.log('3'));
queueMicrotask(() => console.log('4'));
console.log('5');
```

### 2. **Node.js фазы**
Напишите скрипт для Node.js, в котором используются `setTimeout`, `setImmediate` и `process.nextTick`. Покажите, в каком порядке они выполнятся и почему `setImmediate` может выполниться раньше `setTimeout(0)` в определенных условиях.

### 3. **Микрозадачи-паразиты**
Что произойдет, если микрозадача будет бесконечно создавать новые микрозадачи (рекурсивно через `queueMicrotask`)? Сравните это с рекурсией макрозадач через `setTimeout`.
