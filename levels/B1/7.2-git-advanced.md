## |1| **Пул-запросы (Pull Requests)**

В больших проектах никто не пишет код сразу в `main`. Разработка идет в отдельных ветках, а затем создается **Pull Request (PR)** — предложение слить ваш код с основным проектом.

### **Процесс PR:**
1.  **Форк (Fork)** — копия чужого репозитория в ваш аккаунт.
2.  **Ветка (Branch)** — создание новой ветки для вашей задачи.
3.  **Коммит и Пуш** — отправка изменений на GitHub.
4.  **Open PR** — нажатие кнопки "Compare & pull request" на GitHub.
5.  **Ревью (Review)** — коллеги смотрят ваш код, оставляют комментарии и (надеемся) нажимают "Approve".

---

## |2| **Работа над ошибками: Revert & Reset**

Каждый ошибается. Главное — уметь это исправить.

-   **`git commit --amend`** — исправление последнего коммита (например, если забыли добавить файл или опечатались в сообщении).
-   **`git revert [hash]`** — создает НОВЫЙ коммит, который отменяет изменения старого. Это "безопасный" способ, так как он не удаляет историю.
-   **`git reset --soft HEAD~1`** — удаляет последний коммит, но оставляет ваши изменения в коде (они просто перестают быть "закоммиченными").

---

## |3| **Исследование истории: Diff & Blame**

-   **`git diff`** — показывает разницу между текущим кодом и последним коммитом.
-   **`git blame [file]`** — показывает построчно, кто и когда написал каждый символ в файле. Очень полезно, чтобы понять "почему тут так сделано" или кого спросить.

---

## |4| **Введение в CI/CD и GitHub Actions**

**CI/CD** (Continuous Integration / Continuous Development) — это философия автоматизации.

**GitHub Actions** позволяет писать скрипты, которые GitHub будет запускать автоматически.
-   Проверка кода на ошибки (линтеры).
-   Запуск тестов при каждом PR.
-   Автоматический деплой (выкладка на сервер) после слияния веток.

---

## **Итог**

-   **Pull Requests** — основа командной разработки.
-   **Revert** — лучший друг при исправлении ошибок в публичной ветке.
-   **Blame** — инструмент для поиска авторов и контекста кода.
-   **Actions** — способ поручить скучную проверку и выкладку кода самому GitHub.

## **Практика**

### 1. **Исправь соседа (Fork & PR)**
Найдите открытый репозиторий (можно учебный), сделайте Fork, измените один файл (например, исправьте опечатку в README) и создайте Pull Request.

### 2. **Машина времени (Revert)**
Сделайте коммит в свой проект. Затем используйте `git revert`, чтобы отменить его. Посмотрите `git log`, чтобы убедиться, что в истории появились обе записи.

### 3. **Шпионские игры (Blame)**
Откройте любой популярный open-source проект на GitHub. Найдите любой файл с кодом и используйте кнопку **Blame** на сайте. Найдите коммит, который был сделан более 2 лет назад.

### 4. **Первый Action**
Создайте в корне проекта папку `.github/workflows` и файл `main.yml`. Найдите шаблон "Hello World" для GitHub Actions, залейте его и посмотрите во вкладке **Actions** на GitHub, как он успешно выполнится.
