# Правила проекта jsbook

## Структура проекта

- `.agents/` — директория с правилами (rules) и навыками (skills) Граммы.
- Иерархическая структура по уровням: A1, A2, B1, B2, C1, C2
- Каждый уровень имеет директорию с index.md
- Нумерация файлов: `[номер]-[название-темы].md`
- Изображения в `assets/images` с именованием `[номер_раздела]_[описание].ext`

## Форматирование контента

Всегда обращай внимание на правильную нумерацию заголовков и ее наличие там где это необходимо!
Выполняя форматирование никогда не меняй текст заголовка без специального запроса

**Структура документа:**

1. Заголовок первого уровня: `# **Название темы**`
2. Краткое описание (1-3 абзаца). Ключевые технические термины выделяются **жирным** при первом упоминании.
3. Горизонтальная линия `---`
4. Навигационный блок со ссылками
5. Горизонтальная линия `---`
6. Оглавление `## **Содержание**` с нумерованным списком внутренних ссылок. Должно содержать **все** разделы H2 без исключения.
7. Горизонтальная линия `---`
8. Основные разделы
9. Раздел `## **Итог**`. Обязательно предваряется горизонтальной линией `---`.
10. Раздел `## **Практика**`. Обязательно предваряется горизонтальной линией `---`.

**Заголовки:**

- H1: `# **Название**` (жирный текст в звездочках)
- H2 основные разделы: `## |номер| **Название**` (номер в вертикальных черточках, название жирным)
- H3 подразделы: `### номер. **Название**`
- H4 детализация: `#### **номер.номер. Название**`
- H2 итоговые: `## **Итог**`, `## **Практика**`

**Навигационный блок:**

```markdown
- [🏠 Главная](../../readme.md)
- [📚 Все уровни](../index.md)
- [📖 Справочники](../../guides/index.md)
- [🔧 Введение](../../Intro/index.md)
- [⬅️ Предыдущий документ](./06-functions.md)
- [➡️ Следующий документ](../A2/index.md)
```

**Оглавление:**

- Нумерованный список с якорными ссылками
- Формат ссылки: `[**Название раздела**](#номер-название-раздела)`

**Примеры кода:**

- Блоки кода с указанием языка: ```javascript
- Инлайн код в `обратных кавычках`
- Комментарии для объяснения

**Примечания:**

```markdown
> [!Note]
>
> Полный список с описанием методов массивов можно найти [Методы массивов](../../guides/array_methods.md)
```

**Разделители:**

- Горизонтальная линия `---` между основными разделами H2
- Пустые строки для разделения подразделов

**Практический раздел:**

- Структура: `### номер. **Название подраздела**`
- Пронумерованные задачи с примерами ожидаемого результата

## Правила написания

**Код:**

- JavaScript: язык `javascript` или `js`
- HTML: язык `html`
- Результаты в комментариях или отдельно

**Термины:**

- Важные термины выделяются **жирным**
- Англоязычные термины при первом упоминании как ссылки

**Списки:**

- Маркированные списки с `-`
- Нумерованные для последовательных шагов
- Вложенность с 4 пробелами

**Таблицы:**

- Заголовки жирным шрифтом
- Выравнивание по левому краю

## Работа с проектом

**Добавление материалов:**

- Добавлять новые материалы, не удаляя при этом существующие примеры и пояснения, если об этом не просили отдельно.
- Следовать структуре существующих документов
- Обновлять index.md соответствующего уровня
- Проверять работоспособность кода
- В уроках по ESM (модулям) **обязательно** указывать расширение `.js` в путях относительного импорта

**Стиль кода:**

- camelCase для переменных и функций
- const/let вместо var
- Современный ES6+ синтаксис
- Комментарии для сложных участков

**Качество и Педагогика:**

- Проверка кода на современные стандарты
- Понятные объяснения
- **Никогда не удалять** существующие примеры кода или уникальный текст при форматировании или реструктуризации (если это не было явно запрошено).
- Последовательность изложения
- **Визуализация**: Использовать Mermaid для схем и диаграмм в начале сложных разделов (Объекты, Модули, Функции) для создания визуальной ментальной карты.
- **Закрепление материала**: Задачи в разделе `## **Практика**` должны не только проверять новую тему, но и **интегрировать** пройденные ранее темы (циклы, логика, функции) для лучшего усвоения материала.
- Никогда не запускать проверку форматирования через предварительный просмотр
- Использовать markdown для форматирования документов
- Соблюдать единый стиль форматирования во всех документах использую Prettier
- Придерживаться единообразия в оформлении однотипных элементов (заголовков, списков, кода, примечаний)
- Использовать одинаковые форматы примечаний (`[!NOTE]`, `[!WARNING]` и т. д.) во всех документах

## Уроки (усвоено в сессии 2026-03-07)

**Дедупликация контента:**
- Перед добавлением нового раздела **всегда проверять**, не существует ли уже похожий материал в соседних документах. Если одни и те же примеры (`fs`, `path`) присутствуют в двух уроках — они должны быть в одном (более раннем), а во втором — новая тема.
- Правило: **один урок — одна ответственность**. `5.1-nodejs.md` = введение + базовые модули. `5.2-node-modules.md` = система модулей + новые built-in модули.

**Педагогический приоритет:**
- Теория всегда идёт **до** кода, а не после. Даже один абзац "почему это важно" кардинально улучшает восприятие.
- При добавлении раздела «Практика» — задачи должны охватывать **все** новые темы урока, не только первую.

**Структурная осторожность:**
- **Никогда не коммитить** без явного одобрения Харизмы. Ждать команду "коммит".
- При форматировании H2 — сначала читать весь файл, потом делать все замены **одним вызовом** инструмента с несколькими чанками. Не делать множество мелких правок последовательно.

**Навигационные ссылки:**
- **Обязательно** читать `index.md` текущего уровня перед тем, как ставить ссылки «Предыдущий документ» и «Следующий документ».
- Порядок документов определяется **только** файлом `index.md` — никогда не угадывать и не импровизировать.
- `index.md` есть в каждом уровне (`A1/index.md`, `A2/index.md` и т.д.) и является единственным источником правды для структуры раздела.

**Работа с легаси (`old/`):**
- Перед началом работы с любым документом **обязательно проверять** папку `old/` на наличие материала по той же теме.
- Алгоритм: искать по ключевым словам темы (`grep`), прочитать найденный файл, оценить — это реальный контент урока или просто индексный список тем?
- Если найден **реальный контент** (объяснения, примеры кода) — **копировать без искажений** в текущий документ и **удалять из легаси**.
- Если в легаси только индексный список тем (заголовки без тела) — ничего не переносить, темы уже покрыты текущим уроком.
